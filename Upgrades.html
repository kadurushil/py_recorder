Cracking the Code: The TLV Parsing Logic

1

Understanding the Data Stream

The sensor doesn't send clean data. It streams frames, each containing a header and a payload. The payload is made of multiple TLV (Type-Length-Value) blocks.

[Frame] = [Header] + [Payload]
[Payload] = [TLV 1] + [TLV 2] + ...
[TLV] = [Type(4b)] + [Length(4b)] + [Value(Length bytes)]
2

The Breakthrough Moment

The Kotlin app was failing to parse the TLVs correctly. The key insight came from analyzing a working Python script. It revealed how the parsing loop should handle the buffer.

3

The Bug vs. The Fix

The original code forgot to advance the buffer's position past the 'Value' part of the TLV, causing it to read the next header from the wrong place.

Handling Hardware & Android Quirks

Beyond parsing, several other critical issues were resolved:

  • Baud Rate Synchronization: Implemented logic to detect the `baudRate` command from the config and update the app's serial port speed to match the sensor, preventing connection loss.
  • Android Security: Fixed crashes on modern Android versions by correctly specifying receiver export flags (`RECEIVER_NOT_EXPORTED`).
  • Code Stability: Reverted complex changes (like View Binding) back to the original, simple `findViewById` structure to ensure stability and maintain focus on the core logic.
